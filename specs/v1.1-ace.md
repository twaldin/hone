# hone v1.1 — GEPA on dir + ACE on mutator config

## Problem

hone v1 runs a GEPA-style optimization loop over source code: copy dir → mutate → grade → keep if better. This works when the scoring landscape is navigable. In practice, most domains have cliff-like landscapes where the seed is a fragile local maximum and 100% of mutations regress (observed: 0/100 improvements with codex-spark, 0/6+ with gpt-5.4 on hone-a-drone).

The root cause isn't the code loop — it's the mutator's *context*. The playbook, prompt template, and knobs determine what the agent sees and how it reasons. A fixed context produces a fixed failure mode. The system needs to improve its own instructions the way it improves code: through iteration and selection.

## Architecture

Two nested loops:

```
┌─────────────────────────────────────────────────┐
│  ACE outer loop (mutator config optimization)   │
│  Candidates are config dirs, not source dirs    │
│                                                  │
│  ┌─────────────────────────────────────────────┐ │
│  │  GEPA inner loop (source code optimization) │ │
│  │  This is hone v1 — unchanged                │ │
│  │  optimize_repo_frontier() with a policy     │ │
│  └─────────────────────────────────────────────┘ │
│                                                  │
│  Score = best inner score after inner-budget     │
│  iters (or lift over seed, or both — see        │
│  ACE scoring below)                              │
└─────────────────────────────────────────────────┘
```

The inner loop is `optimize_repo_frontier()` — it takes a `MutatorPolicy` and runs `--budget N` mutations on source code. The outer loop evolves the `MutatorPolicy` itself using ACE (Agentic Context Engineering, arxiv 2510.04618).

## Mutator config dir

What ACE evolves. A directory containing three files:

```
config/
├── playbook.md          # mutator instructions (the .md file written to workdir)
├── prompt-template.md   # {variable} template for per-iteration prompts
└── knobs.json           # boolean flags + numeric limits
```

### playbook.md

The agent-facing instructions. Currently `SEED_PLAYBOOK` in policy.py. Contains:
- What to do (edit files to improve grader score)
- How to choose changes (read traces, check recent attempts, targeted edits)
- How to verify (sanity-check before exit)
- Style guidelines

ACE mutates this freely. The only hard rule: it must not contain operational constraints (those come from `OPERATIONAL_CONSTRAINTS`, which is fixed and appended separately).

### prompt-template.md

The per-iteration prompt with `{variable}` placeholders. Currently `SEED_PROMPT_TEMPLATE`. Contains:
- score section (current, best, seed)
- trace_summary / structured_traces
- recent_attempts
- parent_diff_stat / base_diff_stat
- constraints

ACE can: reorder sections, add/remove sections, change section headers, add instructional framing. It cannot remove `{constraints}` or the score block (those are required by the inner loop).

### knobs.json

```json
{
  "include_structured_traces": true,
  "include_recent_attempts": true,
  "include_parent_diff_stat": true,
  "include_base_diff_stat": true,
  "recent_attempts_window": 2,
  "max_trace_chars": 4000,
  "max_trace_summary_chars": 1200
}
```

ACE can toggle booleans and adjust numeric limits. This controls what information reaches the mutator without changing the template.

## ACE outer loop

### Roles (from arxiv 2510.04618)

**Generator** — proposes a new config dir (playbook + template + knobs). Has access to:
- Current frontier configs
- Inner-run traces (which mutations helped/hurt, what the agent did)
- Past ACE attempt history

Generates by modifying one or more of the three config files. This is itself an LLM call — the Generator is a harness adapter reading the config dir and proposing edits.

**Reflector** — analyzes inner-run results after the Generator's config has been scored. Produces a structured reflection:
- What strategies in the config helped
- What strategies hurt
- What was missing
- Patterns across multiple inner-run attempts

The reflection is fed back into the next Generator call as context.

**Curator** — merges, deduplicates, and organizes accumulated insights. Runs periodically (every K ACE iters) to prevent context bloat. The Curator maintains a `reflections.md` file in the ACE run dir that the Generator reads.

### Loop

```
1. Initialize frontier with seed config (current defaults from policy.py)
2. Score seed config: run inner loop (optimize_repo_frontier, inner-budget iters)
3. Record seed_score as baseline
4. For each ACE iteration:
   a. Pick parent from ACE frontier (round-robin, same as inner loop)
   b. Curator: if iter % curate_every == 0, consolidate reflections.md
   c. Reflector: produce reflection on parent's inner-run traces
   d. Generator: propose new config dir, given:
      - parent config
      - reflection
      - reflections.md (curated history)
      - ACE attempt history (what configs were tried, what scores)
   e. Score new config: run inner loop with inner-budget iters
   f. Record score = best inner score (or lift over seed)
   g. Update ACE frontier (same top-N logic as inner)
   h. Emit: ace iter, parent, child, scores, delta
```

### Scoring a config

The ACE score for a config is the **lift over seed** after running the inner loop:

```
ace_score = max(0, inner_best_score - seed_score)
```

If the inner loop finds improvements under this config, the config scores high. If all mutations regress (the v1 failure mode), the config scores 0 — same as seed, no lift.

Why lift and not raw score: raw score is domain-dependent. A config that coasts on a high seed but produces zero improvements is worse than a config that starts from a lower seed but consistently finds gains. Lift measures the config's *effectiveness as an optimizer*.

### Budget

```
hone ace \
  --ace-budget 5 \       # outer iterations
  --inner-budget 20 \    # inner loop iters per ACE iter
  --curate-every 2       # run Curator every N ACE iters
```

Total inner iterations = ace_budget × inner_budget = 100 in this example.

Budget can also be time-based: `--ace-budget-minutes 60` terminates the outer loop after wall-clock time regardless of iterations.

### Generator prompt

The Generator receives a structured prompt:

```
You are optimizing the instructions an AI coding agent receives before it edits code.
Your goal: produce instructions that cause the agent to make BETTER edits —
edits that improve the grader score more often.

Current config (parent):
--- playbook.md ---
{playbook}
--- prompt-template.md ---
{template}
--- knobs.json ---
{knobs}

Reflection on parent's performance:
{reflection}

Curated insights from past iterations:
{reflections_md}

ACE attempt history (configs tried and their scores):
{ace_history}

Produce a NEW config. Modify playbook.md, prompt-template.md, and/or knobs.json.
Output each file in a ``` block with the filename as header.
```

### Reflector prompt

After inner-run completes:

```
Analyze how the mutator agent behaved under this config.

Config:
--- playbook.md ---
{playbook}
--- knobs.json ---
{knobs}

Inner run results:
- Total iterations: {total}
- Improvements: {num_improvements} / {total}
- Best score: {best} (seed was {seed}, lift={lift})
- Score distribution: {histogram}

Sample mutation traces (improvements and regressions):
{sample_traces}

Reflect:
1. What guidance in the playbook helped the agent?
2. What guidance was ignored or counterproductive?
3. What's missing that might help?
4. Were the knobs (trace length, recent attempts window) well-tuned?
5. Any patterns in successful vs failed mutations?
```

## Implementation plan

### Phase 1: Config dir as files

Extract the current hardcoded strings from policy.py into file-based config:

- `MutatorPolicy.from_dir(config_dir: Path) -> MutatorPolicy` — reads playbook.md, prompt-template.md, knobs.json
- `MutatorPolicy.to_dir(config_dir: Path)` — writes them back
- `SEED_CONFIG_DIR` constant pointing to `hone/configs/seed/` bundled with the package

This is a refactor — zero behavioral change. All existing code continues to use `SEED_POLICY` which now loads from the seed dir.

### Phase 2: ACE loop

New file `hone/ace.py`:

- `ace_run()` — outer loop function
- `AceCandidate` — a scored config dir (path, score, parent, reflection)
- `AceFrontier` — top-N config candidates
- `run_reflector()` — LLM call with reflection prompt
- `run_generator()` — LLM call with generator prompt, returns new config dir
- `run_curator()` — LLM call to consolidate reflections.md

Uses harness for all LLM calls. The Generator/Reflector/Curator are harness adapters (default: claude-code:sonnet for Generator, haiku for Reflector and Curator — cost optimization).

### Phase 3: CLI

Add `hone ace` subcommand to cli.py:

```
hone ace \
  --dir <src> \
  --grader <sh> \
  --mutator harness:codex:gpt-5.3-codex \
  --ace-budget 5 \
  --inner-budget 20 \
  --ace-model harness:claude-code:sonnet \     # generator model
  --reflect-model harness:claude-code:haiku \  # reflector model
  --curate-every 2
```

All existing `hone run` flags pass through to the inner loop.

### Phase 4: Storage

ACE run directory layout:

```
<hone-run-dir>/
├── ace/
│   ├── manifest.json            # ace run metadata
│   ├── reflections.md           # curated insights (maintained by Curator)
│   ├── candidates/
│   │   ├── c000-seed/           # seed config (verbatim from policy.py)
│   │   │   ├── playbook.md
│   │   │   ├── prompt-template.md
│   │   │   └── knobs.json
│   │   ├── c001/                # first ACE candidate
│   │   │   ├── playbook.md
│   │   │   ├── prompt-template.md
│   │   │   └── knobs.json
│   │   └── ...
│   ├── reflections/             # per-candidate reflections
│   │   ├── c000.md
│   │   ├── c001.md
│   │   └── ...
│   └── ace-log.jsonl            # per-iteration ACE log
├── inner/                       # inner run dirs (one per ACE candidate)
│   ├── c000/                    # inner run for seed config
│   │   ├── mutations.jsonl
│   │   ├── workdir/
│   │   └── ...
│   ├── c001/
│   └── ...
└── seed/                        # seed snapshot (shared)
```

## Constraints from design principles

1. **Task-agnostic**: playbook and template contain no domain-specific rules. The system discovers domain-specific strategies through the ACE loop (e.g., "on planner.py, targeted parameter edits work better than algorithm rewrites" would be a discovered insight in reflections.md, not a hardcoded constraint).

2. **Self-discovering**: no hardcoded edit granularity, no "only edit constants" rules. The frontier + ACE loop discover what works. If a domain benefits from tiny edits, the Generator will learn to instruct the mutator accordingly. If large rewrites work, it'll learn that too.

3. **Constraints are fixed**: `OPERATIONAL_CONSTRAINTS` (scope, compute budget, time) is never mutated by ACE. Only `playbook_text` is under search.

4. **ACE roles are agents**: Generator, Reflector, Curator are each harness adapter calls. They can use different models. This means ACE itself is harness-agnostic — it works with any adapter.

5. **No recursion**: The outer loop does not call `hone ace` again. ACE optimizes config, GEPA optimizes code. Two layers, no deeper.

## Differences from v1

| | hone v1 | hone v1.1 |
|---|---|---|
| What's optimized | source code only | source code + mutator config |
| Playbook | hardcoded `SEED_PLAYBOOK` | evolved by ACE |
| Prompt template | hardcoded `SEED_PROMPT_TEMPLATE` | evolved by ACE |
| Knobs | hardcoded `PromptKnobs()` defaults | evolved by ACE |
| Budget allocation | `--budget 100` all on code | `--ace-budget 5 --inner-budget 20` |
| Failure mode | 100% regressions → wasted budget | 100% regressions → ACE tries new config |
| Output | best code candidate | best code candidate + best config |
| Cost | budget × per-iter cost | (ace_budget × inner_budget × per-iter) + (ace_budget × 3 LLM calls for G/R/C) |

## Open questions

- **ACE scoring**: lift over seed is simple but noisy (single inner run). Alternatives: median of 3 inner runs, or expected improvement. Ship simple first.
- **Generator freedom**: should the Generator see the full inner-run log or just summaries? Full logs are expensive but more informative. Start with summaries (trace_summary + score histogram).
- **Curator frequency**: every 2 iters is a guess. Too frequent wastes budget; too infrequent lets reflections.md grow unbounded. Make configurable.
- **Restart**: should `hone ace` support `--resume` from an ACE run dir? Likely yes — same pattern as inner loop.
