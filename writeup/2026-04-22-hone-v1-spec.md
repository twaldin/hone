# hone v1 spec

## positioning

`hone` is a repository-state optimizer: an outer search loop that improves a real
codebase by running coding-agent CLI episodes as mutations and grading the
resulting workspace state.

Inspirations / implementation lineage:

- GEPA — frontier search and reflection over evaluation traces
- ACE — future playbook optimization target, not required for v1
- SWE-agent / Aider / Codex / Claude Code — the mutation primitive is a coding
  agent episode in a real workspace

## core loop

1. seed candidate = copy of the target directory
2. grade the seed directory
3. keep a small frontier of best distinct directory states
4. for each iteration:
   - pick a parent from the frontier
   - clone the parent's directory into a temporary workspace
   - inject a tiny playbook file (`CLAUDE.md` or `AGENTS.md`)
   - assemble a minimal prompt from score, traces, recent attempts, and diffs
   - run the coding CLI mutator in that workspace
   - restore/remove the injected playbook so it is not part of the artifact
   - snapshot the resulting directory state
   - grade that snapshot
   - compute parent/base diffs and update the frontier

## artifact semantics

- artifact = repository / directory state on disk
- mutation output = resulting filesystem state after the agent exits
- traces = structured grader feedback and stderr
- provenance = parent diff + base diff + recent-attempt summaries

## v1 seed policy

The seed prompt is intentionally minimal and parameter-heavy. General behavior
lives in a tiny playbook file.

The seed is adapted from GEPA's `optimize_anything` reflection prompt:

- preserve: reflect over evaluation evidence, look for patterns, improve while
  preserving what works
- change: replace text artifact mutation with workspace-state mutation
- move to playbook: generic operating rules like "prefer targeted changes"

## non-goals for v1

- no online ACE observer loop
- no mutator pools / bandits
- no multi-grader Pareto search
- no compose / parallel orchestration
- no public `v1.0.0` tag until the mutator policy itself is meta-optimized on a
  held-out task set

## validation plan

- first validation target: `hone-a-drone`, treated as a holdout-style transfer
  test, not part of policy meta-training
- meta-train policy on other repo tasks across bugfix, performance, and
  reliability domains
- rerun `hone-a-drone` with the meta-optimized policy and compare against:
  - the v0.1 honed controller score
  - the v1 seed-policy run
